# Site Modal Style System

Документ фиксирует единый стиль модальных окон для всего сайта.

Базовым эталоном для этой системы сейчас служат лучшие по качеству модалки из раздела `Кабінет прибиральника -> Мої замовлення`, но сами правила ниже относятся не только к профилю, а ко всем новым и обновляемым модалкам проекта.

## Source Of Truth

Основные файлы-референсы:

- `profile.php`
- `assets/css/cleaners.css`

Текущие эталонные модалки:

- `order-details-modal` из `profile.php`
- `order-action-modal` из `profile.php`

Не использовать как основной референс для новых модалок сайта:

- `orders-modal` из `clean-cemeteries.php`

Это более простая и визуально слабее собранная модалка старого типа. Если где-то на сайте есть похожие старые модалки, их лучше постепенно подтягивать к системе ниже.

## Modal Family

На уровне сайта сейчас есть 2 опорных паттерна:

1. `order-details-modal`
Показывает детальную информацию по заказу.

2. `order-action-modal`
Используется для действий по заказу: принять, отклонить, отправить фото-отчёт.

Обе модалки должны восприниматься как одна дизайн-система. Новые модалки в других разделах сайта должны наследовать этот язык, а не изобретать отдельный визуальный стиль.

## Shared Visual DNA

Общие правила для всех модалок сайта:

- оверлей: `position: fixed`, `inset: 0`, затемнение на тёмно-синей базе, desktop c `backdrop-filter: blur(12px)`
- поверхность: белая карточка с мягкой синей/серой тенью
- рамка: `1px solid rgba(17, 24, 39, 0.08)`
- desktop-радиус: `24px`
- шрифт: `Manrope`
- внутренняя компоновка: `header + scrollable body`
- header визуально лёгкий, с нижней границей
- body имеет светлый градиентный фон, не чисто белый
- иконка и кнопка закрытия визуально равновесны по размеру
- скролл живёт внутри модалки, а не на странице

Базовые desktop-токены:

- ширина details modal: `min(94vw, 760px)`
- ширина action modal: `min(94vw, 680px)`
- compact action modal: `min(94vw, 560px)`
- max-height: `min(88vh, 760px)`
- header padding: `18px 20px 14px`
- body padding: `16px 20px 18px`
- icon/close size: `42px`
- icon/close radius: `14px`

## Desktop Pattern

На desktop модалка должна быть центрирована, а не прилипать к низу экрана.

Визуальные признаки desktop-версии:

- карточка стоит по центру
- вокруг неё есть воздух за счёт `padding: 20px` у overlay
- тень глубокая, но мягкая: `0 28px 72px rgba(15, 23, 42, 0.24)`
- верх и низ карточки полностью скруглены
- footer в action modal липнет к низу контента, но остаётся частью той же карточки

## Mobile Pattern

На mobile это не отдельный дизайн, а тот же компонент, превращённый в `bottom sheet`.

Обязательные mobile-правила:

- модалка открывается снизу, а не по центру
- ширина: `100%`
- max-height: `min(90vh, 760px)`
- радиус только сверху: `28px 28px 0 0`
- боковые и нижняя границы убираются
- overlay без blur, с более лёгким затемнением
- сверху есть drag-handle: полоска `42px x 5px`
- header sticky
- body получает safe-area отступ снизу
- кнопки в footer становятся на всю ширину
- сложные сетки упрощаются до `1 column`

Итог: на mobile должна ощущаться та же модалка, а не новый компонент в другой эстетике.

## Details Modal

`order-details-modal` это эталон для информационных модалок с более презентационной подачей.

Её структура:

- header с иконкой документа, заголовком и кратким подпояснением
- hero-блок заказа
- grid с detail cards
- секции с комментариями и статусными пояснениями
- chips для выбранных услуг
- footer с CTA

Визуальные правила:

- hero-блок: акцентный, но очень светлый
- detail cards: белые, `radius 16px`
- section cards: белые, `radius 18px`
- accent section для комментария клиента: тёплый светлый фон, отдельный визуальный приоритет
- service chips: мягкие синие, не тяжёлые бейджи
- CTA-кнопки в footer: primary + ghost

На mobile:

- hero переходит из 3 колонок в 2
- статус уходит отдельной строкой вниз hero
- grid detail cards становится одноколоночным
- chips услуг становятся в один столбец
- footer-кнопки растягиваются на всю ширину

## Action Modal

`order-action-modal` это эталон для рабочих модалок подтверждения действий.

Её структура:

- header с action-иконкой, заголовком и пояснением
- body из карточек-полей
- sticky footer с primary/secondary action

Визуальные правила:

- modal body не должен выглядеть как голая форма
- каждое логическое действие живёт внутри `order-action-field`
- card field: `radius 18px`, белый фон, мягкая тень
- inputs/textarea/file: `radius 14px`
- quick reasons: карточки выбора, а не обычные radio в строку
- confirm button плотнее и контрастнее cancel button

Цветовая семантика:

- `accept`: синий акцент
- `complete`: синий акцент
- `reject`: розово-красный акцент

Это меняет только акцентные цвета, но не ломает общий модальный стиль.

На mobile:

- action modal тоже становится bottom sheet
- compact/non-compact режим визуально остаются частью одной системы
- footer уходит в колонку
- все кнопки `width: 100%`

## What Must Stay Consistent

Если добавляется новая модалка в любом разделе сайта, она должна наследовать именно этот набор решений:

- тот же overlay-паттерн
- тот же размер и логика header
- тот же размер иконки и close button
- тот же белый surface с мягкой тенью и тонкой рамкой
- тот же light-gradient body
- тот же переход desktop center modal -> mobile bottom sheet
- те же радиусы: `24px` desktop, `28px 28px 0 0` mobile
- тот же типографический режим `Manrope`
- те же принципы sticky header/footer внутри scrollable body

## What To Avoid

- не делать для mobile отдельную тёмную или сильно упрощённую модалку
- не использовать старый плоский стиль `orders-modal` как референс для новых модалок сайта
- не смешивать в одном разделе разные радиусы и разные виды close button
- не делать чисто серые bootstrap-похожие модалки без градиента и внутренних карточек
- не переносить скролл на весь экран, если он уже должен жить внутри модалки
- не делать разные visual language для details modal и action modal

## Practical Rule For Future Work

Перед любыми изменениями модалок по сайту ориентироваться сначала на:

1. `order-details-modal`
2. `order-action-modal`

Если новая модалка не вписывается в их каркас, значит её сначала нужно привести к этой системе, а уже потом развивать дальше.
